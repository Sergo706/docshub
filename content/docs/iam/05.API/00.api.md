All public exports from `@riavzon/auth` are documented below. The module also re-exports several functions from `@riavzon/bot-detector` for convenience.

::warning
Call `configuration()` once before using any export. The entire module reads the resolved config at runtime: calling an export before configuration throws immediately.
::

## Initialization

### `configuration(config)`

Initializes the IAM service. Validates the configuration object against the Zod schema, opens the MySQL connection pools, and resolves the runtime config used by every other export. Call it exactly once at startup.

```ts [server.ts]
import { configuration } from '@riavzon/auth'

await configuration({
  store: { /* ... */ },
  password: { pepper: process.env.PEPPER! },
  botDetector: { enableBotDetector: true },
  htmlSanitizer: { IrritationCount: 50, maxAllowedInputLength: 50000 },
  magic_links: { /* ... */ },
  jwt: { /* ... */ },
  email: { resend_key: process.env.RESEND_KEY!, email: 'no-reply@example.com' },
})
```

See the full [Configuration reference](/docs/iam/configuration) for every option.

### `bootstrapApp(config)`

Wires up the full middleware chain in the correct order and returns a ready-to-use Express app. Available from the `/service` sub-path. This calls `configuration()` internally, so you do not need to call it separately.

```ts [server.ts]
import { bootstrapApp } from '@riavzon/auth/service'

const app = await bootstrapApp(config)
app.listen(10000, '0.0.0.0')
```

The middleware chain applied by `bootstrapApp` is documented in [Getting Started](/docs/iam/getting-started#using-bootstrapapp).

### `startServer()`

Reads the configuration from a JSON file, calls `bootstrapApp` to wire up the full middleware chain, starts listening, and schedules all background data refresh tasks via `refreshData`. The config file path defaults to `/run/app/config.json` and can be overridden with the `CONFIG_PATH` environment variable. After loading, the service deletes the config file from disk unless `SKIP_CONFIG_UNLINK` is set to `true`.

This is the entrypoint used by the Docker image. It is also available from the `/service` sub-path.

```ts [server.ts]
import { startServer } from '@riavzon/auth/service'

await startServer()
```

| Environment variable | Default | Description |
|---------------------|---------|-------------|
| `CONFIG_PATH` | `/run/app/config.json` | Path to the JSON configuration file |
| `SKIP_CONFIG_UNLINK` | `false` | Set to `true` to keep the config file after loading |

### `refreshData(dataSourceInterval, generatorInterval, disposableEmailList)`

Schedules three recurring background tasks that keep the Bot Detector data sources and the disposable email blocklist up to date. Each task runs on a `setTimeout` loop at the specified interval in milliseconds and executes with `nice -n 19` to avoid competing with request handling.

```ts [server.ts]
import { refreshData } from '@riavzon/auth'

// Same intervals used by startServer:
// bot-detector refresh every 24h,
// generate every 3d,
// email list every 7d
refreshData(
  1000 * 60 * 60 * 24,
  1000 * 60 * 60 * 24 * 3,
  1000 * 60 * 60 * 24 * 7,
)
```

| Parameter | Type | Description |
|-----------|------|-------------|
| `dataSourceInterval` | `number` | Milliseconds between `bot-detector refresh` runs |
| `generatorInterval` | `number` | Milliseconds between `bot-detector generate` runs |
| `disposableEmailList` | `number` | Milliseconds between `shield-base --email` runs |

`startServer` calls this automatically after the service starts listening. When using `bootstrapApp` or mounting routes manually, call `refreshData` yourself to keep the data current.


### `scheduleTask(name, cmd, args, interval)`

Schedules a recurring background child process without blocking the event loop. While primarily used internally by `refreshData`, it is exported for general background maintenance tasks.

Commands are executed via `execFile` and prepended with `nice -n 19` to ensure they run at the lowest possible priority. The function uses a recursive `setTimeout` loop rather than a standard `setInterval`. This means that subsequent executions will only queue after the current task resolves, preventing overlapping processes if a command execution time exceeds its designated interval.

To protect the main process from memory exhaustion, the child process buffer is capped at 10MB.

```ts [server.ts]
import { scheduleTask } from '@riavzon/auth'
scheduleTask('my-cleanup', './node_modules/.bin/my-tool', ['--clean'], 1000 * 60 * 60 * 12)
```

| Parameter | Type | Description |
|-----------|------|-------------|
| `name` | `string` | Label used in console log messages |
| `cmd` | `string` | Absolute or relative path to the executable |
| `args` | `string[]` | Arguments passed to the executable |
| `interval` | `number` | Milliseconds to wait after a task completes before running it again |

## Routers

### `authenticationRoutes`

Express `Router` mounting the core authentication endpoints: `POST /signup`, `POST /login`, and `POST /auth/OAuth/:providerName`.

### `tokenRotationRoutes`

Express `Router` mounting the token lifecycle endpoints: `POST /auth/user/refresh-session` and `POST /auth/logout`.

### `magicLinks`

Express `Router` mounting all magic link endpoints: MFA verification, password reset initiation, password reset completion, custom MFA flows, and email update.

### `bffAccessRoute`

Express `Router` mounting `GET /secret/data` and
`GET /secret/accesstoken/metadata`. Both routes run
`requireAccessToken`, `requireRefreshToken`, `getFingerPrint`,
`checkForActiveMfa`, and `protectRoute`; the metadata route also adds
`acceptCookieOnly` before `getAccessTokenPayload`.

### `apiVerificationRoute()`

Factory that returns an Express `Router` mounting `GET /api/public/verify`.
`bootstrapApp()` mounts this router before JSON parsing, cookie parsing, and
the bot-detector middleware because the endpoint only depends on headers, query
parameters, and `req.ip`.

### `apiProtectedRoutes()`

Factory that returns an Express `Router` mounting `POST /api/manage/:action`
and `GET /api/manage/:action` for authenticated API-token management. The POST
route adds `validateContentType('application/json')` and
`express.json({ limit: '1kb' })` after `requireAccessToken`,
`requireRefreshToken`, `getFingerPrint`, `checkForActiveMfa`, and
`protectRoute`. The GET route uses the same session chain without the JSON
guards.

See the [Routes Reference](/docs/iam/api/routes) for the full endpoint
reference with middleware chains and request bodies.

## Middleware

### `protectRoute(req, res, next)`

Verifies the JWT in the `Authorization` header. On failure, responds with `401 Unauthorized`. On success, sets `req.user` with `userId`, `visitor_id`, `accessTokenId` (the JTI, which can be used with `makeRateLimiter()` for per-token rate limiting or early revocation), `roles` (the role array from the token payload), and `payload` (the full decoded JWT object).

```ts [server.ts]
import {
  requireAccessToken,
  requireRefreshToken,
  getFingerPrint,
  checkForActiveMfa,
  protectRoute,
} from '@riavzon/auth'

router.get(
  '/secret/data',
  requireAccessToken,
  requireRefreshToken,
  getFingerPrint,
  checkForActiveMfa,
  protectRoute,
  handler
)
```

### `requireAccessToken(req, res, next)`

Extracts the Bearer token from the `Authorization` header and attaches it to `req.token`. Does not verify or decode the token. Returns `401` if the header is missing or malformed.

### `requireRefreshToken(req, res, next)`

Checks that the `session` cookie containing the refresh token is present. Returns `401` if the cookie is missing. Does not consume or verify the token.

### `checkForActiveMfa(req, res, next)`

Checks `anomaliesCache()` for the hashed refresh token in `req.cookies.session`.
Returns `202` when the session already has a resolvable MFA challenge,
`401` when the cached anomaly requires a full login, and otherwise calls
`next()`.

### `acceptCookieOnly(req, res, next)`

Rejects any request that carries a body, query string, or `Content-Type` header. Ensures the endpoint reads authentication data exclusively from cookies.

### `validateContentType(req, res, next)`

Validates that the request `Content-Type` header is `application/json`. Rejects requests with unexpected content types to prevent CSRF and content-type confusion attacks.

### `verifyMFA(req, res, next)`

Middleware that verifies the email MFA OTP code submitted by the user during the adaptive MFA flow. Used internally by the magic link routes.

### `verifyNewPassword(req, res, next)`

Middleware that validates and hashes the new password submitted during the password-reset flow.

### `customMfaFlowsVerification(req, res, next)`

Middleware that verifies the temporary JWT link for custom MFA flows. Validates the signed token, checks the cache for single-use enforcement, and populates the request with the decoded payload.

### `getFingerPrint(req, res, next)`

Extracts the device fingerprint from the request headers and cookies. Populates the request with fingerprint data used by the anomaly detection pipeline.

See the [Middleware Reference](/docs/iam/api/middlewares) for full details on
every middleware, including parameters, response codes, and behavior.

## Access Tokens

### `generateAccessToken(user)`

Signs and returns a new JWT access token for the given user payload.

```ts
function generateAccessToken(user: AccessTokenPayload): string
```

### `verifyAccessToken(token)`

Verifies an access token through a two-stage process. First, the token must exist in the LRU cache with a valid entry. If the token is not cached, verification fails immediately. Second, regardless of cache presence, the function performs full [JWT](https://datatracker.ietf.org/doc/html/rfc7519) cryptographic verification (signature, algorithm, audience, issuer, subject, and JTI). After signature verification passes, it validates the visitor ID and compares role claims against the cached values.

```ts
function verifyAccessToken(token: string): {
  valid: boolean
  payload?: JwtPayload
  errorType?: string
}
```

### `tokenCache()`

Returns the LRU cache instance that tracks issued access tokens. Tokens are added to this cache at generation time by `generateAccessToken()`, and `verifyAccessToken()` requires a cache entry to proceed. Useful for inspecting cache state or manually invalidating entries (for example, during logout).

### `compareRoles(requiredRaw, providedRaw, log, options?)`

Compares two role arrays for exact set equality. Both arrays must contain the same strings with no extras and no omissions. Duplicate values (before and after normalization) cause a `MalformedPayload` failure. Called internally by `verifyAccessToken` to ensure the roles embedded in the JWT match the roles recorded in the token cache. Export it for use in custom route guards that enforce role requirements beyond what the token inherently carries.

```ts
function compareRoles(
  requiredRaw: unknown,
  providedRaw: unknown,
  log: pino.Logger,
  options?: { normalizer?: (s: string) => string }
): { valid: true } | { valid: false; errorType: 'MalformedPayload' | 'InvalidRoles' }
```

| `errorType` | Cause |
|---|---|
| `MalformedPayload` | Either array is not an array of strings, or contains duplicates before or after normalization |
| `InvalidRoles` | The arrays differ: at least one required role is missing or at least one extra role is present |

## Refresh Tokens

### `generateRefreshToken(ttl, userId, prevTs?)`

Creates a new refresh token with the given TTL, user ID, and optionally, the previous timestamp of the session. The raw token is returned to the caller and the SHA-256 hash is stored in MySQL.

```ts
async function generateRefreshToken(
  ttl: number,
  userId: number
): Promise<IssuedRefreshToken>
```

### `verifyRefreshToken(clientToken)`

Verifies a refresh token against the database without consuming it. Returns the associated user ID, visitor ID, session start time, and expiry.

```ts
async function verifyRefreshToken(clientToken: string): Promise<{
  valid: boolean
  userId?: number
  visitor_id?: string
  reason?: string
  sessionStartedAt?: Date
  expiresAt?: Date
}>
```

### `consumeAndVerifyRefreshToken(clientToken)`

Verifies a refresh token and marks it as consumed in a single atomic operation. Used during token rotation to ensure single-use enforcement. A second call with the same token triggers reuse detection, which revokes all sessions for that user.

```ts
async function consumeAndVerifyRefreshToken(clientToken: string): Promise<{
  valid: boolean
  userId?: number
  visitor_id?: string
  reason?: string
  sessionTTL?: Date  // The session start timestamp (not a duration)
}>
```

### `revokeRefreshToken(clientToken)`

Invalidates a single refresh token by marking it as revoked. The token row remains in the database for audit purposes but can no longer be used. Called during logout and session cleanup.

```ts
async function revokeRefreshToken(clientToken: string): Promise<{ success: boolean }>
```

### `revokeAllRefreshTokens(userId)`

Invalidates every active refresh token for a given user by marking them all as revoked in a single query. Used by reuse detection and account lockout flows to force the user to re-authenticate on all devices.

```ts
async function revokeAllRefreshTokens(userId: string | number): Promise<{ success: boolean }>
```

### `rotateOneUseRefreshToken(ttl, userId, oldClientToken)`

Consumes the old refresh token and issues a new one in a single transaction. The old token becomes invalid immediately.

```ts
async function rotateOneUseRefreshToken(
  ttl: number,
  userId: number,
  oldClientToken: string
): Promise<{ rotated: boolean; raw?: string; expiresAt?: Date }>
```

### `rotateInPlaceRefreshToken(ttl, userId, oldClientToken)`

Updates the existing refresh token row in place with a new hash and expiry. The old raw token becomes invalid but the database row is reused.

```ts
async function rotateInPlaceRefreshToken(
  ttl: number,
  userId: number,
  oldClientToken: string
): Promise<{ rotated: boolean; raw?: string; hashedToken?: string; expiresAt?: Date }>
```

## API tokens

The API-token subsystem exports both the low-level token helpers and the
higher-level management primitives that back the `/api/public/verify` and
`/api/manage/:action` routes.

### `generateChecksum(data)`

Generates the 8-character SHA-256 checksum used in raw API tokens and public
identifiers.

```ts
function generateChecksum(data: string): string
```

### `isSame(a, b)`

Performs a timing-safe comparison between two hexadecimal strings. Used to
compare embedded checksums and reject malformed token or public-identifier
input.

```ts
function isSame(a: string, b: string): boolean
```

### `validateChecksumPublicIdent(publicIdentifier, log)`

Validates the `public_identifier` checksum before the manager performs a
database lookup. Returns `false` when the identifier is malformed or the
checksum does not match.

```ts
function validateChecksumPublicIdent(
  publicIdentifier: string,
  log: pino.Logger
): boolean
```

### `totalUserTokensCount(userId)`

Returns the current valid, invalid, and total API-token counts for one user.
Used internally by creation, metadata, and listing flows, and exported for
dashboard or quota logic.

```ts
async function totalUserTokensCount(userId: number): Promise<{
  userId: number
  totalValidTokens: number
  totalInvalidTokens: number
  total: number
}>
```

### `createApiKey(userId, privilegeType, name, prefix?, expires?, ipAddresses?)`

Creates a new API token row, stores only the SHA-256 digest of the raw token,
and returns the raw API key plus its public identifier once.

```ts
async function createApiKey(
  userId: number,
  privilegeType: 'demo' | 'restricted' | 'protected' | 'full' | 'custom',
  name: string,
  prefix?: string,
  expires?: number,
  ipAddresses?: string[]
): Promise<Results<CreationSuccess>>
```

### `verifyApiKey(rawKey, skipCountUpdates, providedPrivilege, byPassIpCheck, isInternalHash?, ipAddress?)`

Verifies a raw or hashed API token against the database. Enforces checksum
validation, privilege binding, optional IP restrictions, expiration invalidation,
and optional usage-count updates.

```ts
async function verifyApiKey(
  rawKey: string,
  skipCountUpdates: boolean,
  providedPrivilege: 'demo' | 'restricted' | 'protected' | 'full' | 'custom',
  byPassIpCheck: boolean,
  isInternalHash?: boolean,
  ipAddress?: string
): Promise<Results<VerifySuccessResponse>>
```

### `revokeApiKey(rawKey, providedPrivilege)`

Marks an API token as invalid. The direct helper is intentionally idempotent
for invalid or missing tokens.

```ts
async function revokeApiKey(
  rawKey: string,
  providedPrivilege: 'demo' | 'restricted' | 'protected' | 'full' | 'custom'
): Promise<Results<string | {
  msg: string
  invalidedTokenId: number
  userId: number
}>>
```

### `rotateApiKey(rawOldToken, privilegeType, name, deleteOnRotation?, ipAddress?, expires?, prefix?)`

Revokes an existing API token and creates a replacement token. The direct
helper lets you override the new token attributes and optionally delete the old
row after revocation.

```ts
async function rotateApiKey(
  rawOldToken: string,
  privilegeType: 'demo' | 'restricted' | 'protected' | 'full' | 'custom',
  name: string,
  deleteOnRotation?: boolean,
  ipAddress?: string[],
  expires?: number,
  prefix?: string
): Promise<Results<ApiTokenRotationSuccess>>
```

### `getUserApiKeysMetaData(rawApiKey, providedPrivilege)`

Returns token metadata for one API key plus the owner's valid, invalid, and
total token counts. This helper verifies the token without incrementing usage
counters and bypasses IP restrictions.

```ts
async function getUserApiKeysMetaData(
  rawApiKey: string,
  providedPrivilege: 'demo' | 'restricted' | 'protected' | 'full' | 'custom'
): Promise<Results<SingleTokenMeta>>
```

### `updateRestriction(userId, rawToken, newIpAddress)`

Updates the stored IP whitelist for an API token. Pass `null` to remove all IP
restrictions.

```ts
async function updateRestriction(
  userId: number,
  rawToken: string,
  newIpAddress: string[] | null
): Promise<Results<{ msg: string }>>
```

### `updatePrivileges(userId, rawToken, newPrivileges)`

Updates the stored privilege scope for an API token.

```ts
async function updatePrivileges(
  userId: number,
  rawToken: string,
  newPrivileges: 'demo' | 'restricted' | 'protected' | 'full' | 'custom'
): Promise<Results<{ msg: string }>>
```

### `getAllValidTokensList(userId)`

Returns the user's valid API-token inventory plus the overall valid, invalid,
and total token counts.

```ts
async function getAllValidTokensList(
  userId: number
): Promise<Results<AllValidTokensList>>
```

### `privateActionManager(userId, tokenId, publicIdentifier, name, options)`

Coordinates the authenticated API-token management actions. Validates the
public identifier checksum, resolves the matching valid token row, and dispatches
the requested action.

```ts
async function privateActionManager(
  userId: number,
  tokenId: number,
  publicIdentifier: string,
  name: string,
  options: ActionArgs
): Promise<ActionManagerResult>
```

See [API Tokens](/docs/iam/essentials/api) for the full M2M subsystem
documentation.

## Anomaly Detection

### `strangeThings(token, cookie, ipAddress, ua, rotated)`

Runs the nine-check anomaly detection pipeline against the given session context. Returns whether the session is valid, the reason for failure, and whether MFA is required.

```ts
async function strangeThings(
  token: string,
  cookie: string,
  ipAddress: string,
  ua: string,
  rotated: boolean
): Promise<{
  valid: boolean
  reason: string
  reqMFA: boolean
  userId?: number
  visitorId?: string
}>
```

See [Anomaly Detection](/docs/iam/essentials/anomalies) for the full list of checks.

### `anomaliesCache`

Returns the LRU cache instance that tracks the anomalies of an ongoing `session`. Anomalies are added to this cache after a run of `strangeThings`, and keyed by a hashed refresh token. This cache used by [`checkForActiveMfa`](/docs/iam/api/middlewares#checkforactivemfa) and deleted on a successful MFA verification or when errors happens sending an email to a user.

## Password

### `hashPassword(passwords, log)`

Hashes a password using Argon2id with the configured pepper, time cost, memory cost, and hash length.

```ts
async function hashPassword(passwords: string, log: pino.Logger): Promise<string>
```

### `verifyPassword(hash, passwords)`

Verifies a plaintext password against an Argon2id hash.

```ts
async function verifyPassword(hash: string, passwords: string): Promise<boolean>
```

### `isPwned(password)`

Checks a plaintext password against the [Have I Been Pwned](https://haveibeenpwned.com/API/v3#PwnedPasswords) Passwords API using the k-anonymity model. SHA-1-hashes the password, sends the first 5 characters to the API, and checks whether the full hash suffix appears in the response. Results are cached at two levels: the prefix-to-suffix map is cached for 48 hours and individual results for 15 minutes.

```ts
async function isPwned(password: string): Promise<PwnedResponse>

interface PwnedResponse {
  pwned: boolean
  count: number   // number of times the password appeared in known breaches
  date: string    // ISO timestamp of the check
}
```

### `sendTempPasswordResetLink(email)`

Initiates the password-reset flow by sending a magic link email to the given address.

```ts

```

## Controllers

### `apiTokensController(req, res)`

Controller mounted behind `POST /api/manage/:action` and
`GET /api/manage/:action`. Dispatches the API-token creation, listing,
revocation, metadata, rotation, IP update, and privilege update branches after
the authenticated middleware chain has populated `req.user`.

### `verifyApiTokenController(req, res)`

Controller mounted at `GET /api/public/verify`. Reads the raw API token from
the `x-api-key` header, validates the `privilege` query parameter, applies the
API-token verification limiters, and returns the verification result from
`verifyApiKey`.

### `allowBffAccess(req, res, next)`

Controller for the BFF access endpoint. Returns the authenticated user's data to the frontend backend-for-frontend layer.

### `sendOperationalConfig(req, res)`

Controller mounted at `/operational/config`. Returns a subset of the runtime configuration safe for client consumption.

### `getAccessTokenPayload(req, res)`

Controller mounted at `GET /secret/accesstoken/metadata`. Returns the decoded access token payload with TTL information. Reports the time until expiration (`msUntilExp`), a recommended rotation threshold (`refreshThreshold`, 25% of the access token TTL), and a boolean flag (`shouldRotate`) indicating whether the token is within the rotation window. The BFF client uses these fields to proactively refresh tokens before they expire.

### `initCustomMfaFlow(req, res, next)`

Controller that handles `POST /custom/mfa/:reason`. Initiates a custom MFA flow for the given reason.

### `verifyCustomMfa(req, res, next)`

Controller that handles custom MFA verification. Validates the OTP code and completes the custom flow.
